## 1. What changed this pass

### Content (resolves KNOWN_ISSUES.md B1)

- **`/Game/Maps/L_Nets`** — interactive level. DirectionalLight, SkyAtmosphere,
  SkyLight, ExponentialHeightFog, a 100m floor plane, one `CricketPlayerPawn`
  (self-possessing, auto-feeds deliveries), one `PlayerStart`, one `CricketStadium`
  (75 m straight / 68 m square boundary).
- **`/Game/Maps/L_Match`** — auto-playing exhibition level. Same lighting/floor, one
  `CricketMatchRunner` (T20, 20 overs, seed 12345, auto-play on) — the only level
  with a live scoreboard (`CricketHUDDataSource` resolves it from a placed runner).
- **Data assets**: `DA_Ball_RedKookaburra` (`/Game/Data/Balls`), `DA_Team_India` +
  `DA_Team_Australia` (`/Game/Data/Teams`, 11 players each, roles/ratings/pace set).
  These satisfy the registered Asset Manager scans and silence the "0 primary
  assets found" cook warnings; the runtime match itself still hardcodes its squads.

All of the above were created **programmatically**, not by hand in the editor:
`Scripts/setup_content.py` drives Unreal's Python editor-scripting API
(`unreal.EditorLevelLibrary`, `unreal.AssetToolsHelpers`) to build both levels and
all three data assets in one pass. `Scripts/setup_content.sh` is the entry point —
it boots `UnrealEditor` headlessly, runs the script, and verifies the completion
marker in the log. This makes content authoring **repeatable and re-runnable**
(e.g. after a `Content/` wipe) rather than a one-off manual editor session.

### Project configuration

- `CricketSim.uproject` — added `EnhancedInput` (was implicit; checklist called
  for making it explicit), `PythonScriptPlugin`, `EditorScriptingUtilities`
  (Editor-only; required by `setup_content.py`).
- `Config/DefaultGame.ini` — added `+MapsToCook=(FilePath="/Game/Maps/L_Match")`
  alongside the existing `L_Nets` entry.

### Engine-version compatibility (UE 5.7 → 5.8)

This environment has **UE 5.8** installed, not the **UE 5.7** the project is
pinned to (`EngineAssociation` in `.uproject`); 5.7 was not available to install.
Two fixes were required to build/run against 5.8 — both safe, additive, and
inert on a properly matched 5.7 install:

1. **`Source/CricketSim.Target.cs`, `Source/CricketSimEditor.Target.cs`** — added
   `bOverrideBuildEnvironment = true`. UE 5.8 changed the shared-editor-environment
   defaults for three warning levels; without this flag UBT refuses to build an
   Editor-type target against a mismatched shared environment. This is UBT's own
   suggested fix, does not change `DefaultBuildSettings`/C++ standard/any compiled
   behavior, and has no effect when the installed engine already matches.
2. **`Source/CricketPerformance/Private/CricketPerformanceSubsystem.cpp`** — UE 5.8
   removed the public `extern` declaration of the `GGPUFrameTime` global (it's
   still defined engine-side, just no longer exposed via a header). Switched to
   the long-standing accessor function `RHIGetGPUFrameCycles()` (declared in
   `DynamicRHI.h`), which is the supported API and should be available on 5.7 too.

Neither fix touches gameplay/simulation logic. **Recommendation:** re-verify on a
real UE 5.7 install before shipping, since this pass could only validate against 5.8.

## 4. What could *not* be verified in this environment, and why

This pass ran in a sandboxed/headless CLI environment with **no functional
display** (`screencapture` fails with "could not create image from display").
Two specific things are honest gaps, not regressions:

1. **No visual screenshot of the running game.** The packaged app launches and
   runs without crashing (process liveness + CPU activity + absence of crash
   reports all confirm this), but nobody has *looked* at the HUD, the scoreboard,
   the ball, or the camera framing. **Action required:** run
   `Docs/GAMEPLAY_FOOTAGE_CHECKLIST.md` on real hardware with a display attached.
2. **`Saved/Analytics/Matches.csv` did not appear** after the `L_Match` run. This
   matches a pre-existing, already-documented caveat
   (`Docs/CONTENT_AUTHORING_CHECKLIST.md` §9): the packaged app runs under macOS
   App Sandbox (see `Build/Mac/Resources/Sandbox.*.entitlements`), and
   `FFileHelper::SaveStringToFile`'s write may silently no-op outside the
   sandbox container. Confirmed: no CSV anywhere under the app's sandbox
   container (`~/Library/Containers/com.YourCompany.CricketSim/...`) or the
   project tree. **Not a new bug** — the result is checked and logged, never
   crashes — but worth fixing properly (write inside the sandbox container, or
   request a file-access entitlement) before relying on this for telemetry.

Neither gap blocks the milestone: the **rules engine, AI, and full match flow are
already proven by the 122 automated tests run against the real match engine** (in
particular `CricketSim.Match.*` for the rules and `CricketSim.AI.FullMatch` /
`CricketSim.Balance.ValidationReport` for full T20s end-to-end). What's unverified
is purely the *visual/interactive* layer, which automated tests cannot see.
